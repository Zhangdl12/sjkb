# 多看板与数据源选择设计方案

## 1. 设计背景

当前素材分析看板已经完成了第一轮模块化重构，项目从原来的单文件脚本，整理成了 `config`、`data_loader`、`processor`、`metrics`、`ui`、`main` 等分层模块，代码可维护性已经明显提升。

但从下一阶段使用和扩展角度看，当前系统仍然存在两个明显限制：

### 1.1 数据源固定写死

当前应用默认直接读取项目中的固定 Excel 文件，例如：

- `惠氏数据源(新).xlsx`

这种方式在单次分析时问题不大，但当后续由不同人员反复导入新的 Excel 数据时，会带来几个问题：

- 每次更换数据都要手动替换项目内文件；
- 很难确认当前页面实际使用的是哪一个 Excel；
- 试验不同版本数据时不够灵活；
- 对非开发人员不友好，使用门槛偏高。

因此，数据源选择更适合改成页面上传，而不是继续依赖项目内固定文件路径。

### 1.2 入口固定为单看板

当前应用主入口默认只进入一个看板，也就是当前的素材分析总览看板。

随着业务分析需求增加，后续很可能会出现多个分析视角，例如：

- 分类表现看板；
- 渠道表现看板；
- 时间趋势看板；
- SKU 转化看板；
- 爱他美专项分析看板。

如果仍然沿用“一个主入口只对应一个看板”的方式，后面会出现以下问题：

- 主入口逻辑越来越重；
- 多个分析页面容易互相耦合；
- 每次新增看板都要修改主流程；
- 项目结构会逐渐重新退回“堆逻辑”的状态。

因此，下一阶段需要把当前系统升级为：

- 运行时上传数据源；
- 运行时选择要进入的看板；
- 由统一入口分发到不同的 dashboard 页面。

---

## 2. 设计目标

这次方案设计的核心目标如下：

### 2.1 进入应用后先上传数据源

用户启动应用后，不再默认直接读取固定 Excel，而是先通过 Streamlit 上传要分析的 Excel 文件。

推荐使用如下方式：

```python
uploaded_file = st.file_uploader("上传 Excel 文件", type=["xlsx", "xls"])
```

### 2.2 再选择要进入的看板

完成数据源上传后，再让用户选择本次要查看的分析看板。

### 2.3 所有看板默认共用同类 Excel 数据源

当前阶段默认所有看板都共用同一类 Excel 数据源，也就是结构一致的素材明细表。

这样可以保证：

- 数据清洗逻辑尽量复用；
- 指标口径可以共享；
- 新增看板时只关注分析视角，而不用重复造数据读取逻辑。

### 2.4 主入口只负责选择与分发

后续 `main.py` 的职责不再是承载某个具体看板，而是统一负责：

- 文件上传；
- 看板选择；
- 状态控制；
- 加载数据；
- 分发到对应 dashboard。

### 2.5 后续新增新看板时，不重复重写主流程

未来新增一个新看板时，目标是：

- 新建一个 dashboard 文件；
- 注册到统一注册表；
- 复用现有公共数据处理和展示能力；
- 不需要大改主入口。

---

## 3. 整体流程

未来推荐的应用主流程如下：

```text
启动应用
-> 显示 Excel 上传控件
-> 用户上传一个 Excel 文件
-> 校验文件类型与基础结构
-> 列出可用看板
-> 用户选择一个看板
-> 读取已上传 Excel
-> 构建分析宽表
-> 调用对应 dashboard 渲染页面
```

对应到运行方式：

```bash
streamlit run app/main.py
```

进入页面后的推荐交互顺序为：

1. 页面初始化；
2. 显示 `st.file_uploader(...)`；
3. 用户上传 Excel；
4. 显示可用看板下拉框或单选框；
5. 上传成功后加载数据；
6. 进入所选看板页面。

这个流程的重点是：

- 未上传文件前，不加载 Excel；
- 未选择看板前，不进入任何具体分析页面；
- 所有 dashboard 都通过统一入口调度。

---

## 4. 数据源接入规范

为了支持“运行时上传数据源”，后续建议统一采用上传文件的接入规范，而不是目录扫描规范。

### 4.1 上传控件规范

主入口页面统一提供一个 Excel 上传控件：

```python
uploaded_file = st.file_uploader("上传 Excel 文件", type=["xlsx", "xls"])
```

约束建议如下：

- 仅允许上传 `xlsx`、`xls`；
- 未上传文件前，不触发数据读取；
- 上传成功后，再进入后续数据清洗与分析流程。

### 4.2 数据源对象规范

`uploaded_file` 本质上是 Streamlit 返回的上传文件对象，后续数据加载层应支持直接读取该对象，而不是强依赖本地文件路径。

因此建议 `data_loader.py` 后续兼容两类输入：

- 文件路径；
- 上传文件对象。

这样既保留一定兼容性，也方便后续统一迁移到上传模式。

### 4.3 文件命名策略

在上传模式下，不应再依赖固定文件名，例如：

- `惠氏数据源.xlsx`
- `惠氏数据源(新).xlsx`

文件名可以作为页面展示信息或日志信息使用，但不应作为业务逻辑判断依据。真正的可用性判断应基于：

- 文件扩展名是否合法；
- Excel 能否成功读取；
- 必要字段是否齐全。

### 4.4 推荐处理策略

当用户上传文件时，建议按以下方式处理：

- 合法 Excel 进入加载流程；
- 非 Excel 文件直接提示不支持；
- 读取失败时提示“文件无法解析”；
- 字段缺失时提示缺失字段名称；
- 不因为上传了错误文件就导致整个应用崩溃。

## 5. 多看板架构设计

为了支持多个分析页面，建议在现有架构基础上增加“数据源选择模块”和“看板注册模块”。

### 5.1 `main.py` 升级为统一入口

当前 `main.py` 更像是“素材分析总览看板入口”。
未来建议升级为“应用总入口”，负责：

- 初始化页面；
- 上传数据源；
- 选择看板；
- 加载数据；
- 分发到对应 dashboard。

它不再直接承载具体看板的业务细节。

### 5.2 新增 `source_selector.py`

建议新增一个专门负责处理上传数据源状态的模块，例如：

```text
app/source_selector.py
```

该模块的职责建议包括：

- 封装 `st.file_uploader("上传 Excel 文件", type=["xlsx", "xls"])`；
- 判断当前是否已上传文件；
- 返回可供页面展示的数据源描述信息；
- 将上传结果返回给 `main.py` 使用。

也就是说，它只负责“接入和描述数据源”，不负责业务分析。

### 5.3 新增 `registry.py`

新增看板注册模块，例如：

```text
app/registry.py
```

这个模块负责统一维护所有可用看板的注册信息。

每个看板的注册信息建议至少包含：

- `id`：看板唯一标识；
- `name`：页面显示名称；
- `description`：看板简介；
- `render`：看板渲染入口函数。

这样做的目的，是让主入口不需要写死“如果选了 A 就进 A，看了 B 就进 B”的分支结构。

### 5.4 新增 `app/dashboards/`

建议新增专门存放看板页面的目录：

```text
app/dashboards/
```

每个看板单独一个文件，例如：

- `material_dashboard.py`
- `category_dashboard.py`
- `channel_dashboard.py`

每个 dashboard 只负责自己的分析逻辑和页面布局。

### 5.5 每个看板通过统一接口接收所选数据源

建议所有 dashboard 采用统一调用方式。

例如主入口大致遵循这种模式：

```text
上传 Excel
-> 读取数据
-> 构建共享分析宽表
-> 调用目标 dashboard 的 render 函数
```

这样每个看板只需要关心：

- 自己展示什么；
- 自己按什么维度分析；
- 自己需要哪些指标。

而不需要重复写一遍上传控件和入口控制逻辑。

## 6. 推荐目录结构

建议未来把项目逐步演进为下面这种结构：

```text
素材分析看板/
├─ app/
│  ├─ __init__.py
│  ├─ dashboards/
│  │  ├─ material_dashboard.py
│  │  ├─ category_dashboard.py
│  │  └─ channel_dashboard.py
│  ├─ registry.py
│  ├─ source_selector.py
│  ├─ config.py
│  ├─ data_loader.py
│  ├─ processor.py
│  ├─ metrics.py
│  ├─ ui.py
│  └─ main.py
├─ sucai_app.py
├─ 项目设计总结.md
├─ 重构后开发说明.md
└─ 多看板与数据源选择设计方案.md
```

这个结构的优点是：

- `main.py` 只做入口调度；
- `dashboards/` 只放页面；
- `registry.py` 统一维护看板注册；
- `source_selector.py` 统一维护上传数据源接入；
- 原有公共数据链路仍然保留在 `config / data_loader / processor / metrics / ui` 中。

## 7. 后续新增新看板时怎么做

后续每增加一个新的分析看板，推荐按下面流程操作。

### 7.1 新建一个 dashboard 文件

例如你想新增“分类分析看板”，可以新建：

```text
app/dashboards/category_dashboard.py
```

### 7.2 在 `registry.py` 中注册新看板

把这个新看板注册到统一注册表中，填写：

- 唯一 ID；
- 看板名称；
- 简短说明；
- 渲染函数入口。

这样主入口才能在“看板选择列表”里显示它。

### 7.3 复用共享分析宽表

当前建议默认所有看板都复用同一张共享分析宽表。

也就是说：

- 数据读取仍然通过 `data_loader.py`；
- 数据清洗仍然通过 `processor.py`；
- 看板只消费加工后的结果。

如果只是分析视角不同，这样复用最省维护成本。

### 7.4 如果需要新字段，改 `processor.py`

当某个新看板需要新的维度字段、标签字段或派生字段时，优先把公共且可复用的那部分加在 `processor.py`。

例如：

- 新增品牌归类字段；
- 新增投放阶段字段；
- 新增日期周维度字段；
- 新增自定义标签字段。

### 7.5 如果需要新指标，改 `metrics.py`

当某个新看板需要新的汇总指标、比率指标或透视指标时，优先补充到 `metrics.py`。

例如：

- CPC；
- CPA；
- 客单价；
- 分类占比；
- 渠道贡献率。

如果某些指标只有某个看板会用，也可以后续考虑拆分成专属 metrics 文件。

### 7.6 如果需要新页面组件，改 `ui.py` 或专属 UI

如果多个看板都能复用某个页面组件，建议加在 `ui.py`。
如果该组件只属于某个单独看板，可以在该看板文件里做专属 UI，或者新增对应的专属 UI 文件。

### 7.7 主入口不需要为每个看板写死逻辑

新增看板的核心目标之一就是：

- 不再修改 `main.py` 中的大量条件分支；
- 只通过注册表完成挂接；
- 让主入口稳定，新增看板成本低。

## 8. 异常与边界情况

在这个方案落地时，需要提前考虑以下异常场景。

### 8.1 用户未上传文件

如果用户还没有上传 Excel，页面应提示：

- 请先上传 Excel 文件；
- 上传成功后再选择或进入看板。

### 8.2 上传了非 Excel 文件

如果用户上传的不是 `xlsx` 或 `xls`，页面应提示：

- 当前仅支持 Excel 文件；
- 请重新上传正确格式的数据源。

### 8.3 Excel 无法读取

例如文件损坏、内容异常、编码不兼容或不是合法 Excel 容器时，页面应提示：

- 文件解析失败；
- 请检查文件是否完整，或重新导出后再上传。

### 8.4 Excel 结构缺列

即使文件类型正确，Excel 本身也可能缺少某些必要字段。

这类问题仍然应由现有的数据读取 / 数据处理异常机制统一处理，并在页面上友好报错。

### 8.5 某个看板未注册或注册错误

如果看板文件存在，但没有正确注册到 `registry.py`，那么它不应出现在看板选择列表中。
如果注册信息缺失或渲染入口错误，应在入口层捕获并提示，而不是直接报 Python traceback 给用户。

### 8.6 重复上传或切换文件

如果用户重新上传了另一个 Excel，应视为切换数据源，系统需要：

- 重新触发数据加载；
- 清理与旧数据相关的缓存或状态；
- 确保页面展示与当前上传文件一致。

## 9. 推荐实施顺序

为了减少一次性改动过大，建议分阶段实施。

### 第一步：先实现文件上传与选择

先把“固定文件路径”改造成“通过 `st.file_uploader` 上传 Excel”。

这个阶段先不急着做多个看板，只要保证：

- 能正确上传 Excel；
- 能按上传结果完成数据读取；
- 切换上传文件不会出错。

### 第二步：再实现看板注册表

当上传数据源流程稳定后，再加入 `registry.py` 和 dashboard 注册逻辑。

先把当前已有的素材分析总览页面，作为第一个已注册看板接入。

### 第三步：把当前单看板迁移为第一个 dashboard

将现有分析页面从“默认主入口页面”改造成“一个正式的 dashboard 文件”。

这一步完成后，`main.py` 就真正变成“总入口”了。

### 第四步：逐步新增其他分析看板

后续新增分类、渠道、趋势等分析页时，就按照：

- 新建 dashboard；
- 注册；
- 复用共享数据链路；
- 只增加必要的指标与 UI。

这种方式逐步扩展，不容易让项目重新失控。

---

## 10. 设计原则总结

这套方案的核心不是“多写几个页面”，而是建立一套可持续扩展的结构。

整体原则可以总结为：

- 数据源运行时上传；
- 入口统一调度；
- 看板注册化管理；
- 公共数据链路复用；
- 新看板低成本扩展；
- 主入口长期保持稳定。

只要后续开发始终遵守这些原则，项目就可以比较平稳地从“一个素材分析看板”演进成“多个分析看板共存的数据分析平台”。
