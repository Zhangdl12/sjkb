# 素材分析看板开发说明

## 1. 文档目的

这份说明基于当前 `app/` 目录中的实际代码编写，用来帮助后续维护者快速理解：

- 当前应用是如何启动的；
- 数据是如何从上传的 Excel 进入分析流程的；
- `app` 目录下各模块分别负责什么；
- 后续如果要改字段、改指标、改页面，应该优先改哪里。

这是一份面向开发和维护的说明，不再描述已经废弃的“固定本地 Excel 文件作为默认数据源”的旧方案。

---

## 2. 当前项目结构

当前与运行相关的核心结构如下：

```text
素材分析看板/
├─ app/
│  ├─ __init__.py
│  ├─ config.py
│  ├─ data_loader.py
│  ├─ processor.py
│  ├─ metrics.py
│  ├─ ui.py
│  └─ main.py
├─ sucai_app.py
└─ 重构后开发说明.md
```

说明：

- `app/` 是当前正式的应用代码目录。
- `app/main.py` 是当前推荐的主入口。
- `sucai_app.py` 现在只是兼容旧启动方式的薄包装，最终仍然调用 `app.main.main()`。

---

## 3. 当前运行方式

推荐启动命令：

```bash
streamlit run app/main.py
```

兼容旧启动方式：

```bash
streamlit run sucai_app.py
```

但从维护角度，建议统一使用 `app/main.py`。

---

## 4. 当前应用流程

当前应用的执行顺序如下：

1. `main.py` 初始化 Streamlit 页面。
2. 页面显示 `st.file_uploader("上传 Excel 文件", type=["xlsx", "xls"])`。
3. 用户上传 Excel 文件后，`main.py` 读取上传文件的字节内容。
4. `data_loader.py` 负责读取 Excel 中的三张工作表。
5. `processor.py` 将三张原始表加工成一张分析宽表。
6. `ui.py` 渲染侧边栏筛选器。
7. `processor.py` 根据用户选择过滤数据。
8. `metrics.py` 计算顶部汇总指标和透视分析表。
9. `ui.py` 渲染指标卡、透视表、明细表和导出按钮。

可以简化理解为：

```text
上传 Excel
-> 读取三张 sheet
-> 加工成分析宽表
-> 渲染筛选器
-> 按条件过滤
-> 计算指标
-> 展示页面
```

---

## 5. 模块职责

### 5.1 `app/main.py`

`main.py` 是应用主入口，负责组织整个页面流程，但不承载底层业务细节。

当前主要职责：

- 设置页面标题和布局；
- 提供 Excel 上传入口；
- 在上传成功后触发数据加载；
- 捕获数据读取或加工异常并显示错误信息；
- 调用 `processor`、`metrics`、`ui` 完成后续流程；
- 使用 `@st.cache_data` 缓存分析宽表。

当前缓存函数为：

- `load_analysis_dataset(file_bytes: bytes, file_name: str)`

缓存键包含上传文件内容和文件名，因此切换上传文件时会自动失效并重新计算。

### 5.2 `app/config.py`

`config.py` 用于集中维护当前应用的静态约定。

当前主要内容：

- Excel 工作表名称；
- 核心字段名称；
- 清洗失败时的默认填充值；
- 页面支持的筛选字段；
- 默认透视分组字段。

当前不再维护“默认 Excel 文件路径”。

如果后续 Excel 的列名、sheet 名或默认筛选字段发生变化，优先修改这里。

### 5.3 `app/data_loader.py`

`data_loader.py` 只负责“读取数据”，不负责业务清洗。

当前主要职责：

- 接收本地路径、上传文件字节流或文件对象；
- 归一化为 pandas 可读取的数据源；
- 读取三张工作表：
  - `计划类型匹配表`
  - `数据中心-创意数据`
  - `商品匹配表`
- 返回 `SourceTables` 数据对象；
- 在失败时抛出 `DataLoadError`。

这一层的重点是把“Excel 读进来”，而不是做业务处理。

### 5.4 `app/processor.py`

`processor.py` 负责把原始数据加工成可直接分析的宽表，并处理筛选逻辑。

当前主要职责：

- 基于 `推广计划` 生成 `爱他美判断`；
- 关联计划类型映射表，补充：
  - `新产品渠道`
  - `渠道类型`
- 关联商品映射表，补充 `分类`；
- 处理日期列，派生：
  - `年`
  - `月`
  - `日`
- 构建筛选器候选值；
- 根据用户选择过滤数据；
- 在缺列或加工异常时抛出 `DataProcessingError`。

这里是当前应用中最核心的数据清洗层。

### 5.5 `app/metrics.py`

`metrics.py` 负责所有指标口径计算。

当前主要职责：

- 提供 `safe_divide()` 安全除法；
- 计算顶部汇总指标：
  - 展现数
  - 点击数
  - 花费
  - 总订单金额
  - 总订单行
  - CTR
  - CVR
  - ROI
- 构建透视分析表；
- 保持“先聚合，再算比例”的统计口径。

如果后续要新增指标、改公式、改透视汇总方式，优先改这里。

### 5.6 `app/ui.py`

`ui.py` 只负责页面展示，不负责数据读取和业务计算。

当前主要职责：

- 渲染侧边栏筛选器；
- 渲染顶部指标卡；
- 渲染透视分析表；
- 渲染明细预览；
- 提供 CSV 导出按钮；
- 渲染空结果提示。

如果后续要改页面布局、展示文案、控件顺序、表格展示方式，优先改这里。

## 6. 当前数据源方式

当前应用已经统一改成“运行时上传 Excel”模式。

具体表现：

- 未上传文件前，页面只显示上传控件和提示信息；
- 上传后才进入读取和分析流程；
- 不再依赖 `config.py` 中的默认文件名；
- 不再要求项目目录中必须存在某个固定 Excel 文件。

这意味着当前代码的真实数据源入口是：

```python
st.file_uploader("上传 Excel 文件", type=["xlsx", "xls"])
```

而不是本地写死路径。

---

## 7. 当前宽表加工逻辑

上传的 Excel 会被读成三张原始表，然后在 `processor.py` 中合并成一张分析宽表。

当前加工逻辑包括：

1. 从创意数据表中识别 `推广计划` 是否包含关键词 `爱他美`。
2. 生成 `爱他美判断` 字段，值为：
   - `爱他美抢量`
   - `其他`
3. 从计划映射表中按 `计划类型` 关联：
   - `新产品渠道`
   - `渠道类型`
4. 从商品匹配表中按 SKU 关联：
   - `分类`
5. 将 `日期` 从 `YYYYMMDD` 转成时间类型。
6. 进一步派生：
   - `年`
   - `月`
   - `日`

其中：

- `日` 最终格式为 `YYYY/M/D` 字符串；
- 缺失映射值会回填为“未知渠道”“未知渠道类型”“未知分类”。

## 8. 当前筛选与展示逻辑

当前页面的筛选字段定义在 `config.py` 中：

- 时间维度：
  - `年`
  - `月`
  - `日`
- 业务维度：
  - `渠道类型`
  - `新产品渠道`
  - `分类`
  - `爱他美判断`

页面行为如下：

- 默认全选所有候选值；
- 如果某个字段在当前数据中不存在或没有值，相关控件会跳过；
- 当前筛选结果为空时，页面显示空状态提示，不继续渲染指标和表格。

---

## 9. 当前指标口径

当前指标遵循统一口径：

- 先对基础数值列求和；
- 再基于汇总值计算比例类指标。

当前比例定义：

- `CTR = 点击数 / 展现数`
- `CVR = 总订单行 / 点击数`
- `ROI = 总订单金额 / 花费`

这样可以避免逐行先算比例再平均带来的统计失真。

默认透视分析按以下维度分组：

- `新产品渠道`

对应配置位于 `config.py` 的 `default_pivot_group_by`。

---

## 10. 当前导出逻辑

当前明细区支持导出筛选后的结果。

行为如下：

- 页面只预览前 100 行；
- 导出内容是当前筛选后的完整明细；
- 导出格式为 CSV；
- 使用 `utf-8-sig` 编码，以兼容 Excel 打开中文 CSV 的常见场景。

## 11. 后续修改建议

可以按修改目标来判断应该优先改哪个文件：

- 改 sheet 名、列名、默认筛选字段：改 `app/config.py`
- 改 Excel 读取方式：改 `app/data_loader.py`
- 改字段派生、表关联、筛选逻辑：改 `app/processor.py`
- 改指标公式、透视统计：改 `app/metrics.py`
- 改页面布局、控件、表格显示：改 `app/ui.py`
- 改整体页面流程或入口：改 `app/main.py`

---

## 12. 当前维护原则

为了保持当前结构清晰，后续开发建议继续遵守下面几条：

- 不要把数据清洗逻辑重新写回 `main.py`
- 不要在 `ui.py` 中直接写复杂指标计算
- 不要在 `metrics.py` 中做数据读取
- 不要再恢复“写死本地 Excel 路径”的入口模式
- 新增能力时，优先延续 `配置 -> 读取 -> 加工 -> 指标 -> 展示` 的分层方式

---

## 13. 总结

当前项目已经稳定成一个分层明确的 Streamlit 应用：

- `main.py` 负责编排流程；
- `config.py` 负责静态约定；
- `data_loader.py` 负责读 Excel；
- `processor.py` 负责加工和过滤；
- `metrics.py` 负责指标；
- `ui.py` 负责展示。

和旧版本相比，当前最重要的变化是：

- 数据源入口已经从“固定本地 Excel”切换为“页面上传 Excel”；
- 主流程围绕上传文件展开；
- 配置层不再依赖默认文件路径。

后续只要继续按这个分层维护，项目就会比较容易扩展，也不容易再次退回单文件混杂状态。
